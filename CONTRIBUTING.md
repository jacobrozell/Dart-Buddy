# Contributing & code style

This guide documents the conventions the codebase already follows. Match the
surrounding code first; this is the tie-breaker when something is ambiguous.

## Setup

See [`README.md`](README.md#getting-started). The Xcode project is generated by
XcodeGen and is **not** committed — run `xcodegen generate` after pulling
`project.yml` changes.

## Architecture

Layering is one-directional: `App` → `Features` → `Domain` / `Data`. See
[`specs/ArchitectureSpec.md`](specs/ArchitectureSpec.md).

- **Domain** (`Domain/`) is framework-agnostic — no SwiftUI, no SwiftData. Rules
  engines (`X01Engine`, `CricketEngine`, `DartBotEngine`) are pure and unit-tested.
- **Data** (`Data/`, `Persistence/`) hides SwiftData behind repository protocols.
  Features depend on the protocol (`any PlayerRepository`), never the concrete type.
- **Features** use MVVM. The `ViewModel` owns state and orchestrates I/O; the
  `View` is declarative and stateless beyond `@State` for local UI.
- Dependencies are wired once in `AppDependencies` and injected down.

## Swift style

- **Indentation:** 4 spaces, no tabs.
- **No file headers.** Files start with `import` statements — do not add
  author/date/copyright banners.
- **Naming:** `UpperCamelCase` for types, `lowerCamelCase` for members. Booleans
  read as assertions (`isEnabled`, `canStart`, `randomOrder`).
- **Access control:** default to `private`; widen to `internal` only when another
  file needs the symbol. `public` is reserved for the app module's seam types
  (`AppDependencies`, repository protocols, domain entities).
- **Avoid force operations.** No `try!`, `as!`, or `!` unwraps on dynamic values.
  Capture the result of a throwing call in a local instead of force-unwrapping a
  property afterward. (`URL(string:)!` on a verified literal constant is the only
  accepted exception — see `Support/AppLinks.swift`.)
- **No magic numbers** for timing or layout that carries meaning. Name them —
  bot/animation delays live in `Support/BotTurnPacing.swift`, spacing/radius in
  `DesignSystem/Tokens`.
- **Comments explain *why*, not *what*.** Skip doc comments that restate the
  declaration. Reserve `// MARK:` for files with genuinely distinct sections.

## SwiftUI

- Decompose large `body` blocks into `private var section: some View` computed
  properties. When a view file grows past ~400 lines, split a cohesive cluster
  into a `TypeName+Section.swift` **extension file** (see
  `SetupHomeView+OptionChips.swift`) rather than threading bindings through a new
  struct. New files under the target's source folders are picked up automatically
  by XcodeGen.
- Keep async work cancellable: store the `Task` and cancel it in `onDisappear`
  / `cancelPendingWork()`.

## Design system

Two token namespaces, used deliberately — do not mix them on the same surface:

- **`Brand.*`** — the scoreboard palette for Play / Players / Stats / History.
- **`DS.*`** — spacing, radius, and semantic colors for native surfaces (Settings).

See [`DesignSystem/README.md`](DesignSystem/README.md) for the full rule set.

## Accessibility

Interactive controls expose a stable `accessibilityIdentifier` (consumed by
`Tests/UI/WCAGAccessibilityUITests.swift`) and a spoken `accessibilityLabel`.
Prefer Dynamic Type styles over fixed `.font(.system(size:))`. Requirements live
in [`specs/AccessibilitySpec.md`](specs/AccessibilitySpec.md).

## Localization

User-facing strings go through `L10n` and `en.lproj/Localizable.strings` — no
hard-coded display text in views.

## Tests

Add or update tests with behavior changes. All sources live under `Tests/` by layer:

- `Tests/Unit/` — engines, view models, repositories, migrations, policy
- `Tests/Accessibility/` — WCAG contrast and label contracts (Swift Testing)
- `Tests/UI/` — navigation smoke, snapshots, XCTest accessibility audits (UI test target)

Run with `⌘U` or:

```bash
xcodebuild test -scheme DartsScoreboard \
  -destination 'platform=iOS Simulator,name=iPhone 16'
```

## Commits

Write present-tense, imperative subjects that describe the change
("Extract X01 option chips into a dedicated view file"). Keep unrelated changes
in separate commits.
